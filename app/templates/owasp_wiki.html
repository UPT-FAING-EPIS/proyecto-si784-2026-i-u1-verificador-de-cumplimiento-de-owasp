Resumen de Reglas de Seguridad

A continuación se describen las reglas de cumplimiento que el motor analiza de manera automática en cada código fuente, archivo o dirección web, junto con sus penalizaciones y guías para su remediación.

🔐 OWASP-A02: Exposición de Secretos 🔴 CRÍTICA (-30 puntos)

❌ El Problema

Se detectan patrones que corresponden a credenciales o llaves privadas incrustadas directamente (hardcodeadas) en el código fuente:

  • password = "..." o tokens de acceso
  • api_key o api-key expuestas
  • claves secretas de firmas y cifrado

Riesgo: Al publicar el código o distribuirlo, terceros malintencionados pueden extraer estas llaves para vulnerar bases de datos, APIs de pago o servicios en la nube.

✅ La Solución

  • Variables de Entorno: Define variables en un archivo local .env y cargarlas dinámicamente con os.getenv("SECRET") en Python o process.env.SECRET en Node.js.
  • Gestores de Secretos (Vaults): Utiliza servicios dedicados como AWS Secrets Manager, HashiCorp Vault o Azure Key Vault.
  • Excluye del Repositorio: Agrega siempre los archivos de configuración con credenciales (como `.env`) al archivo .gitignore.
📍 Dónde auditar en el motor Componente: app/services/scanner.py verifica expresiones regulares correspondientes a entropía y palabras clave de secretos expuestos.

⚠️ OWASP-A03: Uso de Funciones Peligrosas 🔴 CRÍTICA (-30 puntos)

❌ El Problema

El software invoca subprocesos o intérpretes internos pasando cadenas dinámicas, lo que permite la ejecución remota de comandos (RCE):

  • eval() y exec() con argumentos dinámicos
  • pickle.loads() sobre flujos no confiables
  • subprocess.Popen() con shell=True

Riesgo: Permite que un atacante inyecte sentencias del sistema operativo y tome control absoluto del servidor que hospeda la aplicación.

✅ La Solución

  • No eval/exec: Evita evaluar código a partir de inputs. Usa alternativas seguras como ast.literal_eval() para parsear datos.
  • Subprocesos Seguros: Si ejecutas comandos externos en Python, pasa los argumentos en listas (['ls', '-la']) en lugar de cadenas concatenadas y mantén shell=False.
  • Serialización segura: Usa JSON o Protocol Buffers en lugar de Pickle o YAML no seguros para transmisión de datos.
📍 Dónde auditar en el motor Componente: El parser AST analiza el árbol de sintaxis abstracta para identificar llamadas a funciones peligrosas de la biblioteca estándar de Python y lenguajes populares.

📝 OWASP-A04: Validación de Entrada Insuficiente 🟡 MEDIA (-15 puntos)

❌ El Problema

La aplicación acepta datos desde fuentes externas (HTTP query, parámetros POST, terminal) y los procesa sin realizar un filtrado previo de tipos o límites:

  • Lecturas directas de request.form sin validación
  • Parámetros de URL pasados a queries de DB
  • Uso de variables crudas en sentencias HTML

Riesgo: Abre la puerta a ataques clásicos como inyección SQL (SQLi), Cross-Site Scripting (XSS) y manipulación de rutas de archivos (Path Traversal).

✅ La Solución

  • Esquemas Fuertes: Implementa validaciones automáticas con librerías orientadas a tipos como Pydantic (FastAPI) o Zod (JS).
  • Consultas Parametrizadas: Utiliza ORMs (como SQLAlchemy o Prisma) para prevenir inyecciones SQL en consultas a bases de datos.
  • Listas Blancas (Whitelisting): Valida que los datos coincidan estrictamente con un conjunto conocido de caracteres y longitudes permitidas.
📍 Dónde auditar en el motor Componente: El motor busca rutas HTTP expuestas que reciban argumentos crudos de objetos de request sin la declaración de schemas de validación correspondientes.

🛡️ OWASP-A05: Cabeceras de Seguridad Ausentes 🟡 MEDIA (-15 puntos)

❌ El Problema

Las respuestas del servidor web de producción no incluyen cabeceras HTTP específicas que le indican al navegador del usuario cómo restringir acciones inseguras:

  • Falta Content-Security-Policy (CSP)
  • Falta X-Frame-Options (Clickjacking)
  • Falta Strict-Transport-Security (HSTS)

Riesgo: Mayor susceptibilidad a ataques de Clickjacking (cargar tu web en un frame malicioso) o carga de scripts maliciosos de dominios externos.

✅ La Solución

  • Configura CSP: Define una política restrictiva como default-src 'self' para evitar la carga de scripts no autorizados.
  • Activa HSTS: Fuerce conexiones HTTPS utilizando la cabecera Strict-Transport-Security: max-age=31536000; includeSubDomains.
  • Evita Frames Externos: Añade la cabecera X-Frame-Options: SAMEORIGIN para denegar montajes ajenos.
📍 Dónde auditar en el motor Componente: El escáner de URLs examina las respuestas del servidor HTTP buscando de forma explícita las cabeceras recomendadas por OWASP.

📋 OWASP-A06: Divulgación de Información 🟢 BAJA (-5 puntos)

❌ El Problema

La cabecera de respuesta HTTP expone detalles explícitos de la tecnología que corre por detrás:

  • Server: Apache/2.4.41 (Ubuntu)
  • X-Powered-By: PHP/8.0.3

Riesgo: Facilita a los atacantes el reconocimiento pasivo de versiones precisas de software que contengan vulnerabilidades conocidas (CVEs).

✅ La Solución

  • Ocultar Servidor: En nginx usa server_tokens off;. En Apache usa ServerTokens Prod.
  • Eliminar Cabecera: Remueve la cabecera `X-Powered-By` en tu middleware de aplicación o proxy inverso.
  • Cabeceras Genéricas: Reemplaza el Server por un identificador genérico (ej. "AppGateway").
📍 Dónde auditar en el motor Componente: El analizador HTTP busca patrones de versión y firmas tecnológicas comunes en las cabeceras devueltas en la transacción de red.
