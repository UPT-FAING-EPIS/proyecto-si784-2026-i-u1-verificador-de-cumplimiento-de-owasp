🔐 OWASP-A02: Exposición de Secretos

Severidad: HIGH (-30 puntos)

❌ El Problema:

Se detectaron patrones que corresponden a credenciales o secretos en el código, tales como:

  • password =
  • api_key o api-key
  • secret
  • token

Riesgo: Exponer claves API, contraseñas o tokens en el código fuente permite que atacantes obtengan acceso a sistemas externos o bases de datos.

✅ La Solución:

  1. Usar variables de entorno: Define API_KEY=xxx en tu archivo `.env` y accede via os.getenv("API_KEY")
  2. Usar un Secret Manager: Herramientas como AWS Secrets Manager, Azure Key Vault o HashiCorp Vault
  3. Rotar credenciales: Si ya fueron expuestas, regeneralas inmediatamente
  4. Agregar a .gitignore: Archivos con secretos deben estar en `.gitignore` para no commitearlos

📍 Dónde Corregir:

Archivo: Tu código Python, JavaScript, etc.
Función: analyze_service.py :: execute_scan() busca las coincidencias
Acción: Reemplaza líneas hardcodeadas con os.getenv()

⚠️ OWASP-A03: Uso de Funciones Peligrosas

Severidad: HIGH (-30 puntos)

❌ El Problema:

Se detectaron llamadas a funciones inseguras que pueden permitir ejecución arbuitraria de código:

  • eval() - Evalúa código Python arbitrario
  • exec() - Ejecuta código Python arbitrario
  • pickle.loads() - Deserializa código no confiable
  • subprocess.Popen() - Si recibe entrada del usuario sin sanitizar

Riesgo: Un atacante puede inyectar código malicioso y obtener control total del servidor.

✅ La Solución:

  1. Evita eval() y exec(): Nunca uses estas funciones con entrada del usuario
  2. Usar alternativas seguras: ast.literal_eval() para parsear listas/dicts, no eval()
  3. Para subprocess: Pasa argumentos como lista: subprocess.run(["cmd", arg]), no como string
  4. Para serialización: Usa JSON en lugar de pickle para datos no confiables

📍 Dónde Corregir:

Archivo: Tu código Python
Función: app/services/scanner.py :: scan_code() detecta estas funciones
Acción: Busca y reemplaza eval(), exec(), etc. por funciones seguras

📝 OWASP-A04: Validación de Entrada Insuficiente

Severidad: MEDIUM (-15 puntos)

❌ El Problema:

Se detectó manejo de entrada sin validación explícita:

  • request.args - Parámetros GET sin validar
  • request.form - Datos POST sin validar
  • input() - Entrada del usuario sin verificar

Riesgo: Inyección SQL, XSS, path traversal u otros ataques basados en entrada maliciosa.

✅ La Solución:

  1. Valida tipos: Verifica que los datos sean del tipo esperado (int, string, etc.)
  2. Usa esquemas: Librerías como Pydantic o marshmallow validan automáticamente
  3. Parámetros tipados: En FastAPI, define tipos en los parámetros: def get_item(item_id: int)
  4. Whitelist: Solo acepta valores conocidos y seguros
  5. Sanitiza salida: Escapa caracteres especiales antes de mostrar al usuario

📍 Dónde Corregir:

Archivo: app/routers/analysis.py, app/routers/reports.py
Función: Cualquier ruta que reciba request.args o request.form
Acción: Añade validación con Pydantic en schemas.py o tipos en FastAPI

🔒 OWASP-A05: Cabeceras de Seguridad Ausentes

Severidad: MEDIUM (-15 puntos)

❌ El Problema:

La respuesta HTTP no incluye cabeceras de seguridad críticas:

  • Content-Security-Policy - Previene inyección de scripts
  • Strict-Transport-Security - Fuerza HTTPS
  • X-Frame-Options - Previene clickjacking
  • X-Content-Type-Options - Previene MIME sniffing

Riesgo: Ataques XSS, clickjacking, downgrade a HTTP inseguro.

✅ La Solución:

  1. En FastAPI: Usa middleware para añadir cabeceras
  2. Content-Security-Policy: default-src 'self' para permitir solo recursos del mismo origen
  3. Strict-Transport-Security: max-age=31536000; includeSubDomains
  4. X-Frame-Options: SAMEORIGIN o DENY
  5. X-Content-Type-Options: nosniff

📍 Dónde Corregir:

Archivo: app/main.py
Función: Añade un middleware nuevo o actualiza el existing
Acción: Agrega headers en cada respuesta HTTP

📋 OWASP-A06: Divulgación de Información

Severidad: LOW (-5 puntos)

❌ El Problema:

La respuesta expone la cabecera Server, que revela detalles de la infraestructura:

Server: Apache/2.4.41 (Ubuntu)
Server: nginx/1.18.0

Riesgo: Atacantes pueden identificar versiones y buscar vulnerabilidades conocidas.

✅ La Solución:

  1. Oculta la cabecera Server: Añade a middleware: response.headers["Server"] = "Custom-Server"
  2. Usa nombres genéricos: "Application Server" en lugar de versiones específicas
  3. Configura tu servidor web: nginx: server_tokens off;, Apache: ServerTokens Prod

📍 Dónde Corregir:

Archivo: app/main.py (middleware) o configuración del servidor
Función: Middleware de seguridad
Acción: Elimina o reemplaza la cabecera Server
